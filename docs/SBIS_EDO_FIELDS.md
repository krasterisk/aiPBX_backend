# SBIS EDO field mapping (phase 0 spike)

## `sbisCounterpartyId` — один идентификатор на ответ

В `GET /sbis/counterparty` СБИС отдаёт **не список операторов**, а **одно** поле `sbisCounterpartyId` (в RPC это же значение уходит в `ИдентификаторАЯ` при `ОтправитьПриглашение`).

| Ситуация | Что в ответе |
|----------|----------------|
| СБИС знает контрагента в ЭДО (Saby, Diadoc через роуминг и т.д.) | Один `sbisCounterpartyId`, префикс подсказывает оператора (`2BE` — Saby, `2BM` — Diadoc) |
| Контрагент в другом ОЭД, которого СБИС **не видит** | Поле **пустое** — ID нужно взять у клиента / из кабинета его оператора и ввести вручную в форме |
| Контрагент только в Saby | Часто достаточно ИНН+КПП для приглашения; ID может быть пустым |

**Нет смысла в UI делать выбор «системы ЭДО»** — в API нет массива ящиков, только 0 или 1 id.

## Lookup: два сценария в UI

| Экран | Автоподстановка `sbisCounterpartyId` в поле ID |
|-------|------------------------------------------------|
| Создание **контрагента** tenant (`OrganizationCreateModal`) | **Нет** — клиент вводит ID **своего** оператора при «Подключить ЭДО» |
| **Наши** юрлица admin (`OurOrganizationModal`) | **Да** — подставить из lookup, при необходимости правка вручную |

`GET /sbis/counterparty` по-прежнему отдаёт `sbisCounterpartyId` и `edoOperatorLabel` для админ-формы и отладки.

## Хранение в БД

| Таблица | Колонка (DB / Sequelize) | Содержимое |
|---------|--------------------------|------------|
| `organizations` | `edoParticipantId` | ID контрагента в ЭДО — **только ручной ввод** при «Подключить ЭДО» (не из lookup) |
| `our_organizations` | `edoParticipantId` | ID **нашего** юрлица в ЭДО (можно подставить из lookup в админ-форме) |
| `our_organizations` | `sbisCertThumbprint` | Отпечаток КЭП эмитента для `Сертификат.Отпечаток` в API |

## Префиксы (подсказка `edoOperatorLabel`)

| Префикс | Оператор | Пример |
|---------|---------|--------|
| `2BE` | Тензор / Saby | `2BE72e386a8433e11e38c78005056917125` |
| `2BM` | Контур / Diadoc | `2BM-2465147176-246501001-...` |

## Проверенные ИНН (staging)

См. историю в git; для роуминга СБИС **всё равно** отдаёт один `2BM-…` id, если контрагент известен в маршрутизации СБИС.

## Статус приглашения

| Код | Значение |
|-----|----------|
| 2 | Ожидает принятия |
| 7 | `edoReady` — можно отправлять счета в ЭДО |
| 9 | Маршрут разорван |

## Admin: «Проверить статус» (edit org)

`POST /organizations/:id/edo-invitation/check-route` (только ADMIN):

1. `СБИС.СписокИзмененийПриглашений` — сопоставление по **точному** `edoParticipantId`, ИНН/КПП в ответе и в id (`2BM-…`).
2. Если в списке нет — `СБИС.ОтправитьПриглашение` (probe): для **уже связанных** в Saby СБИС может вернуть JSON-RPC «уже зарегистрирован… можно обмениваться документами» — трактуем как **код 7** без `edoInvitationId`; иначе после отправки — **2** или **7** по `ПрочитатьПриглашение`.

ID вида `2BE…` **не содержит ИНН**, поэтому старый матч только по ИНН в id не работал.

Обычный `…/sync` (tenant, refresh в таблице) **не** вызывает probe — только список / чтение по `edoInvitationId`.
